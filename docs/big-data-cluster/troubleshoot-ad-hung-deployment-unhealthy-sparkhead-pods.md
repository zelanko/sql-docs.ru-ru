# <a name="ad-mode-deployment-hangs---unhealthy-sparkhead-pods"></a>Зависание развертывания в режиме AD — неработоспособные модули pod `sparkhead`

При развертывании BDC в режиме с Active Directory (AD) система перестает отвечать она запросы. Проверьте симптомы, чтобы узнать, является ли причиной отсутствующая запись зоны обратного поиска для контроллера домена в разных сетях узлов кластера.

## <a name="symptom"></a>Симптом

Вы начали развертывать кластер больших данных (BDC) в режиме с AD, но развертывание зависает и не выполняется.

Проверьте работоспособность служб HDFS и Spark. Поищите ошибки в модуле pod `sparkhead`.

## <a name="check-the-hdfs-and-spark-services"></a>Проверка служб HDFS и Spark 

В Azure Data Studio (ADS) подключитесь к контроллеру и просмотрите панель мониторинга кластера больших данных. Убедитесь, что в службах HDFS и Spark есть неработоспособные модули pod `sparkhead`.

![Неработоспособные модули pod sparkhead в службах HDFS Spark](./media/troubleshoot-ad-hung-deployment-unhealthy-sparkhead-pods/hdfs_spark_unhealthy_sparkhead_pods.png)

Извлеките журналы и найдите:

`\mssql-cluster\control-<identifier>\controller\control-<identifier>-controller-stdout.log`.

> [!TIP]
> Собрать журналы можно несколькими способами. Вы можете не копировать журналы с помощью команды [!INCLUDE [azure-data-cli-azdata](../includes/azure-data-cli-azdata.md)], а просто воспользоваться записной книжкой в Azure Data Studio.
> В Azure Data Studio подключитесь к кластеру Kubernetes и запустите соответствующую записную книжку для устранения неполадок. Вот примеры записных книжек:
>
> - TSG027 — наблюдение за развертыванием кластера
> - TSG061 — получение заключительного фрагмента журналов контейнеров для модулей pod в пространстве имен кластера больших данных
> - TSG001 — выполнение `azdata copy-logs`.
>

## <a name="cause"></a>Причина

Нет записи зоны обратного поиска для контроллера домена в записи DNS контроллера домена для сети Kubernetes. В этом примере пропущенная запись — `cni0 10.244`. Контейнеры pod `sparkhead` пытались использовать IP-адрес 10.244.1.30:9000 для доступа к nnode-0-1, но DNS не удалось разрешить его.

:::image type="content" source="media/troubleshoot-ad-hung-deployment-unhealthy-sparkhead-pods/missing_reverse_lookup_zone_entry_for_domain_controller.png" alt-text="Отсутствует запись зоны обратного поиска для контроллера домена":::

## <a name="resolution"></a>Решение

Добавьте недостающую обратную запись DNS (запись типа PTR) для зоны, на которую ссылаются журналы. В этом примере мы добавили 244.10.

:::image type="content" source="media/troubleshoot-ad-hung-deployment-unhealthy-sparkhead-pods/missing_reverse_lookup_zone_entry_for_domain_controller_add.png" alt-text="Добавление отсутствующей записи зоны обратного поиска для контроллера домена":::

> [!NOTE]
> Убедитесь в наличии обратной DNS-записи (записи типа PTR) для самого контроллера домена, зарегистрированного на сервере DNS для всех сетей в узлах кластера.
